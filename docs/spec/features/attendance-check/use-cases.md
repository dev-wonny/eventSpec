# Use Cases

이 문서는 출석체크 기능의 사용자 시나리오와 요구사항 초안을 정리한다. API 목록이 오면 시나리오와 엔드포인트를 1:1 또는 1:N으로 연결한다.

## 액터

- `참여자`: 출석체크를 수행하는 사용자
- `운영자/시스템`: 출석 여부를 확인하거나 상태를 관리하는 주체

## 요구사항 초안

### ATT-REQ-001 출석 가능한 대상을 식별할 수 있어야 한다

- 참여자는 어떤 이벤트 또는 회차에 대해 출석하는지 서버가 식별할 수 있어야 한다.
- 식별 기준은 DDL/API 수신 후 확정한다.

### ATT-REQ-002 참여자는 출석을 요청할 수 있어야 한다

- 참여자는 유효한 입력값으로 출석 요청을 보낼 수 있어야 한다.
- 출석 성공 시 최종 상태를 응답받아야 한다.

### ATT-REQ-003 중복 출석은 통제되어야 한다

- 동일 참여자가 동일 기준 단위에 대해 여러 번 출석 처리되지 않아야 한다.
- 중복 요청 시 정책은 실패 처리 또는 멱등 응답 중 하나로 확정한다.

### ATT-REQ-004 출석 결과를 확인할 수 있어야 한다

- 참여자 또는 운영자는 특정 출석 요청의 처리 결과를 확인할 수 있어야 한다.
- 조회 범위와 응답 항목은 API 목록 수신 후 확정한다.

### ATT-REQ-005 운영 또는 후속 기능이 사용할 출석 기록이 남아야 한다

- 랜덤 게임 등 후속 기능이 참조할 수 있도록 출석 결과가 일관된 형태로 저장되어야 한다.
- 감사 추적이 필요한 경우 관련 컬럼 또는 이력 정책을 함께 정의한다.

## 기본 시나리오

### UC-01 정상 출석

1. 참여자가 출석 가능한 이벤트를 식별한다.
2. 참여자가 출석 요청을 보낸다.
3. 시스템이 입력값과 출석 가능 조건을 검증한다.
4. 시스템이 중복 여부를 확인한다.
5. 시스템이 출석 기록을 저장한다.
6. 시스템이 성공 결과를 응답한다.

### UC-02 중복 출석 시도

1. 참여자가 이미 출석한 대상에 다시 요청한다.
2. 시스템이 중복 여부를 확인한다.
3. 시스템이 정의된 정책에 따라 응답한다.

### UC-03 출석 불가 상태에서 요청

1. 참여자가 종료되었거나 비활성 상태인 대상에 요청한다.
2. 시스템이 상태 또는 시간 조건을 검증한다.
3. 시스템이 출석 불가 응답을 반환한다.

## 비고

- 인증/인가 방식은 아직 미정이다.
- 관리자 전용 조회 API가 필요 없다면 ATT-REQ-004의 범위를 축소할 수 있다.
