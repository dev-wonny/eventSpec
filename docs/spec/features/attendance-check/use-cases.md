# Use Cases

이 문서는 출석체크 기능의 사용자 시나리오와 요구사항을 DDL, 현재 도메인 설명, 확정된 외부 API 계약 기준으로 정리한다.

## 액터

- `참여자`: 출석체크를 수행하는 사용자
- `운영자/시스템`: 출석 여부를 확인하거나 상태를 관리하는 주체

## 요구사항 초안

### ATT-REQ-001 출석 가능한 대상을 식별할 수 있어야 한다

- 외부 서버는 `GET /event/v1/events/{eventId}`로 출석 이벤트 전체 회차와 상태를 조회할 수 있어야 한다.
- 참여자는 어떤 이벤트와 어떤 일자성 회차에 대해 출석하는지 서버가 식별할 수 있어야 한다.
- `event_round`는 출석 일자 단위로 해석한다.
- 조회 API는 `eventId`를 Path로 받고, `X-Member-Id`가 있으면 개인 상태를 함께 반환한다.

### ATT-REQ-002 참여자는 출석을 요청할 수 있어야 한다

- 외부 서버는 `POST /event/v1/events/{eventId}/rounds/{roundId}/entries`로 출석 요청을 보낼 수 있어야 한다.
- 요청에는 `X-Member-Id`가 포함되어야 한다.
- `event_applicant`는 사전 참여 가능 대상자 테이블이 아니라 회차별 applicant 기준 테이블로 해석해야 한다.
- 이번 출석체크에서는 회원별 사전 참여 가능 대상 체크를 두지 않는다.
- `event_round.event_id`는 FK로 보호되더라도, 서버는 요청 `eventId`와 `round.event_id`가 일치하는지 Service에서 검증해야 한다.
- 출석 요청 시 서버는 `event_applicant`를 `round_id + member_id` 기준으로 한 건만 생성해야 한다.
- 출석체크는 회차당 보상 매핑이 최대 1개다.
- 출석 성공 시 `event_entry`는 `event_id`, `applicant_id`, `member_id`를 저장하고 회차는 applicant를 통해 해석한다.
- 추첨형 이벤트에서는 같은 회차에도 여러 `event_entry`가 들어갈 수 있고, 최초 `is_winner = false`에서 나중에 update될 수 있다.
- 출석체크형 이벤트는 즉시 보상이므로 `event_entry.is_winner = true`로 저장한다.
- 회차에 point 보상 매핑이 있으면 로컬 당첨/보상 확정 이력이 `event_win`에 남아야 한다.
- 회차에 보상 매핑이 없으면 point는 지급하지 않고 `event_win`도 생성하지 않는다.
- `event_applicant.round_id`는 FK로 보호하고, applicant-이벤트 값참조 정합성은 Service에서 검증해야 한다.

### ATT-REQ-003 중복 출석은 통제되어야 한다

- 동일 참여자는 동일 `roundId + memberId` 기준으로 `event_applicant`가 한 번만 생성되어야 한다.
- 출석체크 이벤트에서는 오늘 날짜 회차가 `roundId`가 되므로, 같은 날짜 재요청은 같은 `roundId + memberId` 재요청이다.
- 다른 날짜의 회차에는 다시 출석할 수 있어야 한다.
- 중복 요청 시 출석 실패로 처리하고, 프론트에는 `이미 출석했습니다`를 노출한다.

### ATT-REQ-004 출석 결과를 확인할 수 있어야 한다

- 외부 서버는 `GET /event/v1/events/{eventId}`로 전체 회차 목록과 출석 상태를 확인할 수 있어야 한다.
- `X-Member-Id`가 있으면 `ATTENDED / MISSED / TODAY / FUTURE` 상태와 보상 정보를 반환해야 한다.
- `X-Member-Id`가 없으면 상태와 win 정보 없이 기본 회차 정보만 반환해야 한다.

### ATT-REQ-005 운영 또는 후속 기능이 사용할 출석 기록이 남아야 한다

- 랜덤 게임 등 후속 기능이 참조할 수 있도록 참여 이력은 `event_entry`, 로컬 당첨/보상 확정 이력은 `event_win`에 일관되게 저장되어야 한다.
- 감사 추적이 필요한 경우 관련 컬럼 또는 이력 정책을 함께 정의한다.

### ATT-REQ-006 외부 보상 지급 연동은 로컬 트랜잭션과 분리되어야 한다

- 회차에 point 보상 매핑이 있는 출석체크는 `event_applicant`, `event_entry`, `event_win`을 먼저 로컬 트랜잭션으로 저장해야 한다.
- 외부 point API는 로컬 트랜잭션 커밋 후 호출해야 한다.
- 외부 point API 호출이 실패하거나 응답이 없더라도 로컬 출석 이력은 롤백하지 않는다.
- 외부 point API 실패는 로그와 운영 알림으로 처리한다.
- 회차에 보상 매핑이 없으면 외부 point API를 호출하지 않는다.

## 기본 시나리오

### UC-01 정상 출석

1. 참여자가 출석 가능한 이벤트를 식별한다.
2. 참여자가 특정 회차에 대한 출석 요청을 보낸다.
3. 시스템이 이벤트, 회차, 시간 조건을 검증한다.
4. 시스템이 `round.event_id == event.id`인지 검증한다.
5. 시스템이 같은 `round_id + member_id` 기준으로 `event_applicant` 생성을 바로 시도한다.
6. 같은 키의 applicant가 이미 있으면 중복 출석으로 종료한다.
7. applicant 생성이 가능하면 해당 회차의 applicant를 저장한다.
8. 시스템이 회차에 연결된 보상 매핑이 있는지 확인한다.
9. 시스템이 `event_entry`를 저장한다.
10. 보상 매핑이 있으면 시스템이 `event_win`을 저장하고 로컬 트랜잭션을 커밋한다.
11. 로컬 트랜잭션 커밋 후 외부 point API를 호출한다.
12. 외부 API가 실패하거나 무응답이면 시스템이 로그를 남기고 운영 알림 대상으로 기록한다.
13. 시스템이 성공 결과를 응답한다.

### UC-02 다른 날짜 재출석

1. 참여자가 이미 다른 날짜에 출석한 같은 이벤트에 다음 날짜 회차로 다시 요청한다.
2. 시스템이 다음 회차에 대한 `event_applicant`를 새로 생성한다.
3. 시스템이 새 회차의 보상 매핑 존재 여부를 확인한다.
4. 시스템이 새로운 회차에 대한 `event_entry`를 저장한다.
5. 보상 매핑이 있으면 새로운 회차에 대한 `event_win`까지 저장하고 로컬 트랜잭션을 커밋한다.
6. 커밋 후 외부 point API를 호출하고, 실패 시 로그와 운영 알림으로 처리한다.
7. 시스템이 성공 결과를 응답한다.

### UC-03 동일 날짜 중복 출석 시도

1. 참여자가 이미 출석한 같은 회차 또는 같은 날짜 기준으로 다시 요청한다.
2. 시스템이 같은 `round_id + member_id` 기준으로 applicant를 insert하려고 시도한다.
3. 시스템이 unique 충돌을 중복 출석으로 판단한다.
4. 시스템이 프론트에 `이미 출석했습니다` 오류를 반환한다.

### UC-04 출석 불가 상태에서 요청

1. 참여자가 운영 중단되었거나 참여가 마감된 이벤트 또는 회차에 요청한다.
2. 시스템이 상태 또는 시간 조건을 검증한다.
3. 시스템이 출석 불가 응답을 반환한다.

### UC-05 외부 point API 실패 또는 무응답

1. 참여자가 유효한 출석 요청을 보낸다.
2. 시스템이 이벤트/회차, applicant 생성 시도, 보상 매핑을 확인한다.
3. 시스템이 `round.event_id == event.id`인지 검증한다.
4. 시스템이 `event_entry`, `event_win`을 저장하고 로컬 트랜잭션을 커밋한다.
5. 시스템이 외부 point API를 호출한다.
6. 외부 API가 실패하거나 응답하지 않는다.
7. 시스템이 로컬 데이터를 롤백하지 않고, 오류 로그와 운영 알림을 남긴다.
8. 출석 응답은 성공으로 유지하고 후속 운영 보정 대상으로 관리한다.

### UC-06 무보상 출석

1. 참여자가 유효한 출석 요청을 보낸다.
2. 시스템이 이벤트/회차와 applicant 생성을 검증한다.
3. 시스템이 해당 회차에 보상 매핑이 없음을 확인한다.
4. 시스템이 외부 point API를 호출하지 않는다.
5. 시스템이 `event_entry`만 저장한다.
6. 시스템이 무보상 출석 성공 결과를 응답한다.

## 비고

- 인증/인가 방식은 아직 미정이다.
- 현재 외부 API는 2개만 제공하고, admin 기능은 범위 밖이다.
- `event_applicant`는 별도 관리 API로 적재하지 않고 출석 요청 시 생성한다.
- DB 스키마는 `promotion`을 사용하고, 시간 컬럼은 `TIMESTAMP`로 관리하며 애플리케이션 기준 타임존은 `Asia/Seoul`이다.
- 현재는 외부 보상 API를 같은 요청 처리 안에서 동기 호출하지만, 로컬 트랜잭션 커밋 이후 단계로 분리한다.
