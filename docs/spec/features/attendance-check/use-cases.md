# Use Cases

이 문서는 출석체크 기능의 사용자 시나리오와 요구사항을 DDL, 현재 도메인 설명, 확정된 외부 API 계약 기준으로 정리한다.

## 액터

- `참여자`: 출석체크를 수행하는 사용자
- `운영자/시스템`: 출석 여부를 확인하거나 상태를 관리하는 주체

## 요구사항 초안

### ATT-REQ-001 출석 가능한 대상을 식별할 수 있어야 한다

- 외부 서버는 `GET /event/v1/events/{eventId}`로 출석 이벤트 전체 회차와 상태를 조회할 수 있어야 한다.
- 참여자는 어떤 이벤트와 어떤 일자성 회차에 대해 출석하는지 서버가 식별할 수 있어야 한다.
- `event_round`는 출석 일자 단위로 해석한다.
- 조회 API는 `eventId`를 Path로 받고, `X-Member-Id`가 있으면 개인 상태를 함께 반환한다.

### ATT-REQ-002 참여자는 출석을 요청할 수 있어야 한다

- 외부 서버는 `POST /event/v1/events/{eventId}/rounds/{roundId}/entries`로 출석 요청을 보낼 수 있어야 한다.
- 요청에는 `X-Member-Id`가 포함되어야 한다.
- `event_applicant`는 실제 참여 기록이 아니라 참여 가능 대상자 풀로 해석해야 한다.
- 출석 요청 시 서버는 `event_applicant`에서 `event_id + member_id` 기준 대상을 식별하고, `round_id`가 지정된 경우 요청 회차와도 정합해야 한다.
- 출석체크는 회차당 보상 매핑이 최대 1개다.
- 출석 성공 시 `event_entry`가 `event_id`, `round_id`와 함께 append-only로 저장된다.
- 회차에 point 보상 매핑이 있으면 point 지급 성공 이력이 `event_win`에 남아야 한다.
- 회차에 보상 매핑이 없으면 point는 지급하지 않고 `event_win`도 생성하지 않는다.

### ATT-REQ-003 중복 출석은 통제되어야 한다

- 동일 참여자는 동일 `eventId + roundId + memberId` 기준으로 한 번만 출석 처리되어야 한다.
- 출석체크 이벤트에서는 오늘 날짜 회차가 `roundId`가 되므로, 같은 날짜 재요청은 같은 `eventId + roundId + memberId` 재요청이다.
- 다른 날짜의 회차에는 다시 출석할 수 있어야 한다.
- 중복 요청 시 출석 실패로 처리하고, 프론트에는 `이미 출석했습니다`를 노출한다.

### ATT-REQ-004 출석 결과를 확인할 수 있어야 한다

- 외부 서버는 `GET /event/v1/events/{eventId}`로 전체 회차 목록과 출석 상태를 확인할 수 있어야 한다.
- `X-Member-Id`가 있으면 `ATTENDED / MISSED / TODAY / FUTURE` 상태와 보상 정보를 반환해야 한다.
- `X-Member-Id`가 없으면 상태와 win 정보 없이 기본 회차 정보만 반환해야 한다.

### ATT-REQ-005 운영 또는 후속 기능이 사용할 출석 기록이 남아야 한다

- 랜덤 게임 등 후속 기능이 참조할 수 있도록 참여 이력은 `event_entry`, 실제 지급 보상 이력은 `event_win`에 일관되게 저장되어야 한다.
- 감사 추적이 필요한 경우 관련 컬럼 또는 이력 정책을 함께 정의한다.

### ATT-REQ-006 외부 보상 지급 연동은 원자적으로 처리되어야 한다

- 회차에 point 보상 매핑이 있는 출석체크는 point 보상 지급이 성공한 경우에만 최종 성공으로 처리되어야 한다.
- 외부 point API 호출이 실패하거나 응답이 없으면 `event_entry`, `event_win`은 모두 롤백되어야 한다.
- 회차에 보상 매핑이 없으면 외부 point API를 호출하지 않고 `event_entry`만 저장한다.
- 현재는 프론트가 즉시 실패를 인지할 수 있도록 동기 응답 기반으로 처리한다.

## 기본 시나리오

### UC-01 정상 출석

1. 참여자가 출석 가능한 이벤트를 식별한다.
2. 참여자가 특정 회차에 대한 출석 요청을 보낸다.
3. 시스템이 이벤트, 회차, 시간 조건을 검증한다.
4. 시스템이 `event_applicant`에서 해당 사용자의 참여 가능 대상 여부를 확인한다.
5. `event_applicant.round_id`가 존재하면 요청 `roundId`와 일치하는지 검증한다.
6. 시스템이 해당 회차 기준 중복 출석 여부를 확인한다.
7. 시스템이 회차에 연결된 보상 매핑이 있는지 확인한다.
8. 보상 매핑이 있으면 외부 point API를 호출한다.
9. 보상 매핑이 있고 외부 API가 성공하면 `event_entry`와 `event_win`을 커밋한다.
10. 보상 매핑이 없으면 `event_entry`만 커밋한다.
11. 시스템이 성공 결과를 응답한다.

### UC-02 다른 날짜 재출석

1. 참여자가 이미 참여 가능 대상자로 등록된 이벤트에 다음 날짜 회차로 다시 요청한다.
2. 시스템이 동일 이벤트의 `event_applicant` eligibility를 다시 확인한다.
3. 시스템이 새 회차의 보상 매핑 존재 여부를 확인한다.
4. 보상 매핑이 있으면 외부 API 성공 후 새로운 회차에 대한 `event_entry`와 `event_win`을 저장한다.
5. 보상 매핑이 없으면 새로운 회차에 대한 `event_entry`만 저장한다.
6. 시스템이 성공 결과를 응답한다.

### UC-03 동일 날짜 중복 출석 시도

1. 참여자가 이미 출석한 같은 회차 또는 같은 날짜 기준으로 다시 요청한다.
2. 시스템이 같은 `event_id + round_id + member_id` 기준의 기존 출석 이력을 확인한다.
3. 시스템이 중복 출석으로 판단한다.
4. 시스템이 프론트에 `이미 출석했습니다` 오류를 반환한다.

### UC-04 출석 불가 상태에서 요청

1. 참여자가 종료되었거나 비활성 상태인 이벤트 또는 회차에 요청한다.
2. 시스템이 상태 또는 시간 조건을 검증한다.
3. 시스템이 출석 불가 응답을 반환한다.

### UC-05 외부 point API 실패 또는 무응답

1. 참여자가 유효한 출석 요청을 보낸다.
2. 시스템이 참여 가능 대상자 여부, 회차, 보상 매핑을 확인한다.
3. 시스템이 외부 point API를 호출한다.
4. 외부 API가 실패하거나 응답하지 않는다.
5. 시스템이 `event_entry`, `event_win`을 모두 롤백한다.
6. 프론트에는 현재 출석체크가 불가능하다는 오류를 반환한다.

### UC-06 무보상 출석

1. 참여자가 유효한 출석 요청을 보낸다.
2. 시스템이 참여 가능 대상자 여부와 회차를 검증한다.
3. 시스템이 해당 회차에 보상 매핑이 없음을 확인한다.
4. 시스템이 외부 point API를 호출하지 않는다.
5. 시스템이 `event_entry`만 저장한다.
6. 시스템이 무보상 출석 성공 결과를 응답한다.

## 비고

- 인증/인가 방식은 아직 미정이다.
- 현재 외부 API는 2개만 제공하고, admin 기능은 범위 밖이다.
- 현재 프로젝트는 `event_applicant` 관리 API를 제공하지 않으므로 대상자 데이터는 SQL 또는 별도 admin 프로젝트에서 관리한다.
- 현재는 보상 매핑이 있는 경우에만 외부 보상 API가 동기식 성공해야 출석 성공으로 확정된다.
