# Use Cases

이 문서는 출석체크 기능의 사용자 시나리오와 요구사항을 DDL과 현재 도메인 설명 기준으로 정리한다. API 목록이 오면 시나리오와 엔드포인트를 1:1 또는 1:N으로 연결한다.

## 액터

- `참여자`: 출석체크를 수행하는 사용자
- `운영자/시스템`: 출석 여부를 확인하거나 상태를 관리하는 주체

## 요구사항 초안

### ATT-REQ-001 출석 가능한 대상을 식별할 수 있어야 한다

- 참여자는 어떤 이벤트와 어떤 일자성 회차에 대해 출석하는지 서버가 식별할 수 있어야 한다.
- `event_round`는 출석 일자 단위로 해석한다.
- 클라이언트가 `round_id`를 직접 넘길지, 서버가 현재 시각으로 활성 회차를 찾을지는 API 목록 수신 후 확정한다.

### ATT-REQ-002 참여자는 출석을 요청할 수 있어야 한다

- 참여자는 유효한 입력값으로 출석 요청을 보낼 수 있어야 한다.
- 첫 출석이면 `event_applicant`를 생성하고, 이후에는 기존 참여자 앵커를 재사용해야 한다.
- 출석 성공 시 `event_entry`가 append-only로 저장되고, point 지급 성공 이력이 `event_win`에 남아야 한다.
- 현재 출석 성공은 외부 point API 호출 성공까지 포함한다.

### ATT-REQ-003 중복 출석은 통제되어야 한다

- 동일 참여자가 동일 회차 또는 동일 출석일 기준으로 여러 번 출석 처리되지 않아야 한다.
- 오늘 날짜 기준으로 이미 `event_entry`가 존재하면 이미 출석한 것으로 본다.
- 다른 날짜의 회차에는 다시 출석할 수 있어야 한다.
- 중복 요청 시 출석 실패로 처리하고, 프론트에는 `이미 출석했습니다`를 노출한다.

### ATT-REQ-004 출석 결과를 확인할 수 있어야 한다

- 참여자 또는 운영자는 특정 이벤트/회차 기준 출석 처리 결과를 확인할 수 있어야 한다.
- 조회 범위와 응답 항목은 API 목록 수신 후 확정한다.

### ATT-REQ-005 운영 또는 후속 기능이 사용할 출석 기록이 남아야 한다

- 랜덤 게임 등 후속 기능이 참조할 수 있도록 참여 이력은 `event_entry`, 실제 지급 보상 이력은 `event_win`에 일관되게 저장되어야 한다.
- 감사 추적이 필요한 경우 관련 컬럼 또는 이력 정책을 함께 정의한다.

### ATT-REQ-006 외부 보상 지급 연동은 원자적으로 처리되어야 한다

- 출석체크는 point 보상 지급이 성공한 경우에만 최종 성공으로 처리되어야 한다.
- 외부 point API 호출이 실패하거나 응답이 없으면 `event_entry`, `event_win`은 모두 롤백되어야 한다.
- 현재는 프론트가 즉시 실패를 인지할 수 있도록 동기 응답 기반으로 처리한다.

## 기본 시나리오

### UC-01 정상 출석

1. 참여자가 출석 가능한 이벤트를 식별한다.
2. 참여자가 특정 회차에 대한 출석 요청을 보낸다.
3. 시스템이 이벤트, 회차, 시간 조건을 검증한다.
4. 시스템이 기존 `event_applicant` 존재 여부를 확인한다.
5. 첫 참여자면 `event_applicant`를 생성하고, 기존 참여자면 재사용한다.
6. 시스템이 해당 회차 기준 중복 출석 여부를 확인한다.
7. 시스템이 회차에 연결된 point 보상 정책을 확인한다.
8. 시스템이 외부 point API를 호출한다.
9. 외부 API가 성공하면 `event_entry`와 `event_win`을 커밋한다.
10. 시스템이 성공 결과를 응답한다.

### UC-02 다른 날짜 재출석

1. 참여자가 이미 참여자 앵커가 존재하는 이벤트에 다음 날짜 회차로 다시 요청한다.
2. 시스템이 기존 `event_applicant`를 재사용한다.
3. 시스템이 새 회차의 point 보상 정책으로 외부 API를 호출한다.
4. 외부 API 성공 후 새로운 회차에 대한 `event_entry`와 `event_win`을 저장한다.
5. 시스템이 성공 결과를 응답한다.

### UC-03 동일 날짜 중복 출석 시도

1. 참여자가 이미 출석한 같은 회차 또는 같은 날짜 기준으로 다시 요청한다.
2. 시스템이 오늘 날짜 기준 기존 `event_entry`를 확인한다.
3. 시스템이 중복 출석으로 판단한다.
4. 시스템이 프론트에 `이미 출석했습니다` 오류를 반환한다.

### UC-04 출석 불가 상태에서 요청

1. 참여자가 종료되었거나 비활성 상태인 이벤트 또는 회차에 요청한다.
2. 시스템이 상태 또는 시간 조건을 검증한다.
3. 시스템이 출석 불가 응답을 반환한다.

### UC-05 외부 point API 실패 또는 무응답

1. 참여자가 유효한 출석 요청을 보낸다.
2. 시스템이 참여자, 회차, 보상 정책을 확인한다.
3. 시스템이 외부 point API를 호출한다.
4. 외부 API가 실패하거나 응답하지 않는다.
5. 시스템이 `event_entry`, `event_win`을 모두 롤백한다.
6. 프론트에는 현재 출석체크가 불가능하다는 오류를 반환한다.

## 비고

- 인증/인가 방식은 아직 미정이다.
- `event_entry`가 회차를 직접 저장하지 않기 때문에, 조회와 중복 판단의 기준을 API/서비스 규칙으로 더 명확히 정해야 한다.
- 현재는 외부 보상 API가 동기식 성공해야만 출석 성공으로 확정된다.
- 관리자 전용 조회 API가 필요 없다면 ATT-REQ-004의 범위를 축소할 수 있다.
