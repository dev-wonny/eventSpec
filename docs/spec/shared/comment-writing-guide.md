# Comment Writing Guide

## 1. 목적

본 표준은 소스코드 내 주석의 작성 기준을 통일하여 코드의 이해도, 유지보수성, 협업 효율성을 향상시키는 것을 목적으로 한다.

본 표준은 다음 사항을 달성하기 위해 적용한다.

- 코드의 책임과 역할을 빠르게 이해할 수 있도록 한다.
- 코드만으로 드러나지 않는 의도, 제약사항, 예외 조건, 외부 연동 배경, 동작 규칙을 명확히 남긴다.
- 쉽게 낡는 정보를 코드 밖으로 분리하여 주석의 신뢰도를 유지한다.
- 주석을 코드와 동일한 수준으로 관리하고 리뷰한다.

## 2. 적용 범위

본 표준은 다음 대상에 적용한다.

- Java, Python, JavaScript 등 사내 개발 언어 전반
- 백엔드, 프론트엔드, 배치, 운영 스크립트, 공통 라이브러리
- 신규 개발 코드 및 유지보수 대상 기존 코드

## 3. 용어 정의

### 3.1 문서 주석

클래스, 인터페이스, 메서드, 함수 등 선언부에 작성하는 주석으로, 사용하는 개발자가 알아야 하는 동작 규칙을 설명하는 주석을 의미한다.

### 3.2 인라인 주석

구현 내부 특정 로직에 대해 작성하는 주석으로, 구현 이유, 제약사항, 외부 시스템 보정, 예외 처리 배경 등을 설명하는 주석을 의미한다.

### 3.3 동작 규칙

입력 조건, 반환 규칙, 실패 조건, 제한사항, 상태 전이 규칙, 트랜잭션 경계 등 호출자 또는 유지보수자가 반드시 알아야 하는 실행 규칙을 의미한다.

### 3.4 외부 문서

아키텍처 문서, 시퀀스 다이어그램, 운영 가이드, 위키 등 코드 외부에서 관리되는 문서를 의미한다.

## 4. 기본 원칙

### 4.1 현재 기준으로 유효한 사실만 작성한다

주석은 현재 코드 이해에 필요한 정보만 작성한다.

과거 변경이력, 작성자, 버전 증가 이력 등 시간이 지나면 쉽게 낡는 정보는 코드 주석에 남기지 않는다.

### 4.2 코드에 드러나는 내용은 반복하지 않는다

클래스명, 메서드명, 변수명, 타입, 반환형, 애너테이션만 봐도 알 수 있는 내용은 주석으로 반복하지 않는다.

주석은 다음 내용을 우선 설명해야 한다.

- 왜 이 로직이 필요한가
- 어떤 조건에서 동작하는가
- 어떤 입력 조건이 필요한가
- 어떤 결과를 반환하는가
- 어떤 경우 실패하거나 예외가 발생하는가

### 4.3 문서 주석과 인라인 주석의 목적을 구분한다

문서 주석은 사용하는 개발자가 알아야 하는 동작 규칙을 설명한다.

인라인 주석은 구현 이유, 우회 처리, 외부 시스템 제약, 예외 처리 배경을 설명한다.

### 4.4 주석은 짧고 구조적으로 작성한다

장황한 설명보다 다음 구조를 기본으로 한다.

- 요약 1문장
- 필요 시 보충 설명 1~3줄
- `@param`, `@return`, `@throws`
- 필요한 경우 동작 규칙 또는 구현 주의사항

### 4.5 주석은 코드와 함께 유지보수한다

코드가 변경되면 관련 주석도 반드시 함께 수정한다.

코드와 불일치하는 주석은 없는 주석보다 더 위험하므로 제거하거나 즉시 갱신한다.

## 5. 작성 기준

### 5.1 주석에 포함할 정보

주석에는 다음 정보를 포함할 수 있다.

- 비즈니스 규칙
- 입력 조건
- 반환 규칙
- 예외 발생 조건
- `null` 허용 여부
- 부작용 여부
- 보안상 주의사항
- 성능상 이유
- 외부 시스템 제약
- DTO 매핑 여부와 이유
- 원시 응답 여부
- 상태 전이 규칙
- 트랜잭션 경계가 중요한 경우 그 이유
- 임시 우회 처리 이유와 제거 조건

### 5.2 주석에 포함하지 않을 정보

주석에는 다음 정보를 포함하지 않는다.

- 작성자명
- 작성일
- 수정일
- 버전 번호
- 변경이력
- 클래스명 또는 메서드명 반복
- 보면 알 수 있는 타입 설명
- 화면 기준 사용 위치
- 전체 호출 흐름
- 시스템 전체 시퀀스
- 의미가 불명확한 표현

코드에는 현재 동작과 동작 규칙만 남기고, 변경이력·작성자·운영 배경·호출 흐름 같은 정보는 Git, PR, API 문서, 위키로 분리한다.

금지 예시는 다음과 같다.

- 임시 처리
- 중요
- 버그 수정
- 나중에 변경

## 6. 유형별 작성 표준

### 6.1 클래스 주석

#### 6.1.1 작성 대상

다음 클래스에는 문서 주석 작성을 권장한다.

- Controller
- 외부 노출 Service / Facade
- 배치 Job
- 외부 연동 Client
- 정책 클래스
- 핵심 도메인 처리 클래스

#### 6.1.2 작성 내용

클래스 주석에는 다음 내용을 포함한다.

- 클래스의 책임
- 제공 기능 범위
- 필요한 경우 외부 연동 제약
- 필요한 경우 주요 동작 특성

#### 6.1.3 작성 제외 항목

다음 항목은 클래스 주석에 작성하지 않는다.

- `author`
- `version`
- `since` 작성일
- 수정이력
- 상세 호출 흐름
- FE/BE 전체 시퀀스 설명

#### 6.1.4 예시

```java
/**
 * partner-admin 회원 조회 API를 처리하는 Controller.
 *
 * 회원 목록 조회와 상세 조회 기능을 제공한다.
 * 요청값 검증 후 조회 유스케이스에 위임하며, 응답은 ApiResponse 형식으로 반환한다.
 *
 * 구현 주의사항:
 * 상세 조회 응답 구조는 내부 시스템 스키마의 영향을 받을 수 있다.
 */
@RestController
@RequestMapping("/partner-admin/members")
public class MemberController {
}
```

### 6.2 메서드 주석

#### 6.2.1 작성 목적

메서드 주석은 메서드를 호출하는 개발자가 알아야 하는 다음 정보를 설명해야 한다.

- 입력 조건
- 반환 규칙
- 실패 조건
- 제한사항
- 필요한 경우 동작 규칙

#### 6.2.3 작성 기준

메서드 주석은 다음 기준을 따른다.

- 요청/응답 타입명을 그대로 반복하지 않는다.
- 시그니처에 이미 드러나는 정보는 다시 설명하지 않는다.
- 타입보다 의미를 설명한다.
- 사용 위치는 원칙적으로 작성하지 않는다.
- 비동기 타입 사용 여부보다 완료/실패 방식이 더 중요하다.
- 원시 응답 여부, DTO 변환 여부, 오류 처리 방식처럼 호출자가 알아야 할 규칙을 우선 설명한다.

#### 6.2.4 일반 예시

```java
/**
 * 회원 번호를 기준으로 회원 상세 정보를 조회한다.
 *
 * 요청값의 회원 번호는 필수이며, 조회 결과를 표준 응답 형식으로 반환한다.
 * 조회 대상이 없거나 요청값이 올바르지 않으면 오류로 처리한다.
 *
 * @param requestDto 회원 상세 조회 조건
 * @return 회원 상세 정보
 * @throws IllegalArgumentException 회원 번호가 누락된 경우
 */
public Mono<ApiResponse<PartnerMemberResDto>> getMemberDetail(
        PartnerMemberReqDto.DetailReqDto requestDto) {
    ...
}
```

#### 6.2.5 원시 응답 예시

```java
/**
 * 회원 상세 정보를 조회한다.
 *
 * 내부 시스템의 상세 응답을 별도 DTO로 변환하지 않고 원시 형태로 반환한다.
 * 응답 필드 구조는 내부 시스템 변경의 영향을 직접 받을 수 있다.
 *
 * @param requestDto 회원 상세 조회 조건
 * @return 회원 상세 원시 응답
 * @throws IllegalArgumentException 필수 조회 조건이 누락된 경우
 */
public ApiResponse<Map<String, Object>> getMemberDetailRaw(
        PartnerMemberReqDto.DetailReqDto requestDto) {
    ...
}
```

### 6.3 메서드 주석에서 트랜잭션 및 도메인 흐름 작성 기준

#### 6.3.1 기본 원칙

트랜잭션의 상세 절차나 도메인의 전체 흐름은 메서드 주석에 장문으로 작성하지 않는다.

다만, 메서드의 이해에 직접 필요한 수준의 트랜잭션 경계와 도메인상 중요한 동작 순서는 제한적으로 작성할 수 있다.

#### 6.3.2 작성 가능한 내용

다음은 메서드 주석에 포함할 수 있다.

- 하나의 트랜잭션으로 묶여야 하는 이유
- 읽기 전용 여부
- 실패 시 전체 롤백 여부
- 커밋 이후 수행되는 처리
- 상태 전이 순서가 중요한 이유
- 순서가 바뀌면 안 되는 비즈니스 규칙

#### 6.3.3 작성하지 않는 내용

다음은 메서드 주석에 작성하지 않는다.

- 전체 레이어 호출 체인
- 화면에서 시작되는 전체 흐름
- 자주 바뀌는 내부 처리 절차
- 단순 구현 순서를 번호로 나열한 설명

#### 6.3.4 예시

```java
/**
 * 주문 취소를 처리한다.
 *
 * 결제 취소, 재고 복원, 주문 상태 변경을 하나의 트랜잭션으로 처리한다.
 * 처리 중 하나라도 실패하면 전체 작업을 롤백한다.
 *
 * @param command 주문 취소 요청 정보
 * @throws InvalidOrderStateException 취소 가능한 상태가 아닌 경우
 */
public void cancelOrder(CancelOrderCommand command) {
    ...
}

/**
 * 배송 완료를 확정한다.
 *
 * 배송 상태 변경과 이력 저장은 하나의 트랜잭션으로 처리한다.
 * 알림 이벤트는 커밋 이후에만 발행한다.
 *
 * @param deliveryId 배송 식별자
 */
public void completeDelivery(Long deliveryId) {
    ...
}
```

### 6.4 인라인 주석

#### 6.4.1 좋은 예

```java
// 내부 시스템이 탈퇴 회원을 ACTIVE로 반환하는 경우가 있어,
// 파트너 어드민 정책에 맞게 상태를 별도로 보정한다.
status = adjustMemberStatus(rawStatus, deletedAt);
```

#### 6.4.3 나쁜 예

```java
// 상태 보정
status = adjustMemberStatus(rawStatus, deletedAt);
```

### 6.5 TODO / FIXME / HACK

#### 6.5.1 공통 규칙

`TODO`, `FIXME`, `HACK` 주석은 다음 기준을 따른다.

- 담당 팀, 담당자, 이슈 번호 중 하나 이상 포함
- 제거 조건 또는 완료 조건 포함
- 모호한 표현 금지

#### 6.5.2 예시

```java
// TODO(ITPJT-1234): 회원 상세 응답을 표준 DTO로 전환한 뒤 Map 반환 제거
// FIXME(member-api): 시스템 API가 null 대신 빈 객체를 반환하면 방어 로직 제거
// HACK: 레거시 phoneNumber 포맷 불일치로 인해 임시 정규화 수행. 응답 포맷 통일 후 제거
```

## 7. 표준 템플릿

### 7.1 클래스 주석 템플릿

```java
/**
 * [도메인/기능명] 관련 기능을 처리하는 [클래스 유형].
 *
 * [주요 기능 1], [주요 기능 2]를 제공한다.
 * 필요 시 [외부 연동 제약 / 정책상 주의점 / 주요 동작 특성]을 기술한다.
 */
```

### 7.2 메서드 주석 템플릿

```java
/**
 * [무엇]을 수행한다.
 *
 * [입력 기준, 처리 방식, 반환 방식]을 설명한다.
 * 필요 시 [원시 응답 여부, DTO 매핑 여부, 제한사항]을 기술한다.
 *
 * @param [파라미터명] [의미]
 * @return [반환값의 의미]
 * @throws [예외 타입] [발생 조건]
 */
```

### 7.3 트랜잭션/도메인 규칙 포함 템플릿

```java
/**
 * [무엇]을 수행한다.
 *
 * [핵심 처리 내용]을 설명한다.
 * [트랜잭션 경계 또는 상태 전이 규칙]이 중요할 경우 그 이유만 간략히 기술한다.
 *
 * @param [파라미터명] [의미]
 * @return [반환값의 의미]
 * @throws [예외 타입] [발생 조건]
 *
 * 동작 규칙:
 * - [규칙 1]
 * - [규칙 2]
 */
```

### 7.4 인라인 주석 템플릿

```java
// [이 처리가 필요한 이유]
// [정책상 또는 외부 시스템상 배경]
```

### 7.5 TODO / FIXME / HACK 템플릿

```java
// TODO([이슈번호/팀명]): [해야 할 작업] 후 [제거 또는 전환 조건]
// FIXME([이슈번호/팀명]): [현재 문제]가 해결되면 [제거 대상]
// HACK: [임시 우회 이유]. [정상화 조건]
```
